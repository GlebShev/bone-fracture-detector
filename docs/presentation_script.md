# Сценарий видеопрезентации (3–4 минуты)

## 0:00–0:25 — задача

«Я сделал учебный сервис детекции возможных переломов на рентгеновских снимках. Пользователь
загружает изображение, выбирает быструю или более точную модель и получает локализацию.
Сервис не является медицинским диагностическим инструментом».

## 0:25–0:55 — данные

Показать `data_audit.json`, два-три снимка с ground-truth boxes и объяснить переход от
семи несбалансированных исходных классов к одному продуктовому классу `fracture`. Назвать
реальные размеры split: 3 631 / 348 / 169 изображений.

## 0:55–1:30 — тюнинг и результаты

Показать `metrics.csv`:

- Fast — YOLO11n, 640 px;
- Accurate — YOLO11s, 768 px;
- `mAP@0.5`, precision, recall и latency;
- Fast: `mAP@0.5 = 0.154`, Accurate: `0.149`;
- явно сказать, что порог 0.5 из ТЗ не достигнут и модельная часть даёт 1 балл.

Не читать все гиперпараметры. Сказать, что гипотеза о преимуществе большей модели не
подтвердилась: Fast немного точнее и примерно втрое быстрее. Объяснить, почему
агрессивные цветовые аугментации отключены для X-ray.

## 1:30–1:55 — backend

Открыть <https://bone-fracture-detector-xpw8.onrender.com/docs>, показать `GET /models`,
затем один `POST /predict`. В двух фразах: валидация файла, ленивая загрузка модели,
структурированный JSON и размеченный PNG.

## 1:55–3:05 — frontend

Открыть <https://jcvh3mrbarmvekrnoybn6d.streamlit.app/>. После долгого бездействия заранее
разбудить бесплатный Render запросом к `/health`, чтобы не ждать холодный старт в записи.

1. Загрузить снимок.
2. Выбрать Fast, выполнить запрос, показать boxes/confidence/latency.
3. Не меняя снимок и threshold, выбрать Accurate и сравнить.
4. Изменить confidence и показать влияние порога.
5. Показать корректное сообщение при отсутствии детекций.

## 3:05–3:35 — ошибки и ограничения

Показать по одному false positive и false negative. Объяснить влияние маленьких областей,
контраста и domain shift. Повторить, что confidence не равен вероятности медицинского
диагноза.

## 3:35–3:50 — финал

Показать GitHub и публичную ссылку. Коротко сформулировать, почему по текущему test-набору
предпочтительнее Fast, а Accurate оставлена как второй пользовательский режим и
отрицательный результат тюнинга.
