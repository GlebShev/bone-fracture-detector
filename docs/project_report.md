# Отчёт: сервис детекции переломов

## 1. Постановка задачи

Цель проекта — создать продуктовый сервис, который локализует возможные переломы на
рентгеновском снимке верхней конечности. Пользователь выбирает быстрый или более точный
режим, загружает изображение и получает bounding boxes, классы, confidence и latency.

Сервис является учебной демонстрацией и не используется для постановки диагноза.

## 2. Данные

Источник: `pkdarabi/bone-fracture-detection-computer-vision-project` на Kaggle, лицензия
CC BY 4.0. Выбрана detection-версия разметки YOLO, в которой каждая строка имеет формат
`class_id x_center y_center width height` с нормализованными координатами.

Перед обучением выполняется автоматический аудит:

- читаемость изображений;
- диапазоны координат и площадь boxes;
- соответствие ID списку классов;
- пустые и отсутствующие labels;
- распределение классов;
- SHA-256 дубликаты между train/validation/test.

Аудит подготовленной detection-выборки дал 4 148 изображений и 2 388 объектов:

| Split | Images | Boxes | Empty labels |
|---|---:|---:|---:|
| Train | 3 631 | 2 088 | 1 827 |
| Validation | 348 | 204 | 175 |
| Test | 169 | 96 | 86 |

Битых изображений, невалидных координат и одинаковых файлов между split не найдено.
Исходный класс `humerus fracture` имеет только 3 объекта в train и отсутствует в
validation/test. Поэтому до обучения была зафиксирована продуктовая одноклассовая
постановка: все 2 388 исходных областей сохранены, их class ID заменён на `fracture`.

## 3. Модели и тюнинг

**Fast** — YOLO11n, вход 640 px, 35 эпох. Это компактный режим для минимальной задержки.

**Accurate** — YOLO11s, вход 768 px, 50 эпох. Гипотеза эксперимента: большая модель и
разрешение смогут лучше локализовать небольшие или слабоконтрастные области.

Обе модели инициализируются предобученными весами. Seed равен 42, используется early
stopping. Для медицинских изображений отключены изменения hue/saturation, vertical flip,
mosaic и mixup. Оставлены небольшие геометрические преобразования и изменение яркости.

## 4. Протокол оценки

Гиперпараметры выбираются только по validation split. Test split запускается один раз после
фиксации моделей. Основная метрика из ТЗ — `mAP@0.5`; цель максимального балла — не менее
0.5. Дополнительно фиксируются `mAP@0.5:0.95`, precision, recall, время инференса, число
параметров и размер весов.

| Model | mAP@0.5 | mAP@0.5:0.95 | Precision | Recall | Inference, ms | Size, MB |
|---|---:|---:|---:|---:|---:|---:|
| Fast | 0.1536 | 0.0428 | 0.2985 | 0.1979 | 111.1 | 5.19 |
| Accurate | 0.1487 | 0.0411 | 0.2601 | 0.1868 | 338.7 | 18.27 |

Метрики получены на 169 test-снимках (96 размеченных областей) одним запуском
`scripts/evaluate.py` на CPU. Ни одна модель не достигла порога 0.5 из ТЗ. Fast немного
лучше по всем четырём quality-метрикам и примерно втрое быстрее. Гипотеза о преимуществе
YOLO11s/768 на этой выборке не подтвердилась.

## 5. Архитектура продукта

```text
Browser → Streamlit → POST /predict → FastAPI → ModelManager → selected YOLO
                                                ↓
                     JSON detections + annotated PNG (base64)
```

Backend валидирует MIME-тип, объём и размеры изображения. Модели загружаются лениво и один
раз на процесс. Для каждой модели есть отдельный lock, поэтому параллельные запросы не
портят состояние inference runtime. Frontend обрабатывает пустой результат, ошибки API и
недоступность весов.

## 6. Анализ ошибок

В `reports/figures/` сохранены батчи ground truth и предсказаний обеих моделей. В них
наблюдаются:

- корректная локализация;
- false positive;
- false negative;
- пропуски небольших и слабоконтрастных областей.

Вероятные причины: малый размер перелома, слабый контраст, большое число снимков без
объектов, ограниченный объём разметки и domain shift между источниками снимков.

## 7. Вывод

Порог `mAP@0.5 ≥ 0.5` не достигнут, поэтому модельная часть соответствует 1 баллу по
критериям. Fast — предпочтительный режим: `mAP@0.5 = 0.154`, 111 мс и 5.19 МБ. Accurate
оставлена в интерфейсе как обязательная вторая самостоятельно выбираемая модель и как
результат проверенной, но не подтвердившейся гипотезы тюнинга. Для улучшения качества
нужны ревизия разметки, дополнительные снимки с переломами и отдельный цикл экспериментов.
