# Отчёт по проекту

## Задача

Проект решает одноклассовую задачу детекции переломов на рентгеновских снимках верхних
конечностей. На вход поступает изображение, на выходе сервис возвращает bounding boxes,
confidence и размеченный PNG. В интерфейсе можно выбрать одну из двух моделей и изменить
порог уверенности.

## Данные

Источник: Kaggle-датасет `pkdarabi/bone-fracture-detection-computer-vision-project`,
лицензия CC BY 4.0. Исходные сегментационные полигоны преобразованы в минимальные
axis-aligned bounding boxes.

| Split | Images | Boxes | Empty labels |
|---|---:|---:|---:|
| Train | 3 631 | 2 088 | 1 827 |
| Validation | 348 | 204 | 175 |
| Test | 169 | 96 | 86 |

Перед обучением проверены:

- читаемость изображений;
- диапазоны координат и площадь boxes;
- соответствие class ID списку классов;
- пустые и отсутствующие labels;
- SHA-256-дубликаты между train, validation и test.

Битых изображений, невалидных координат и дубликатов между split не найдено. В исходной
выборке семь классов, но один из них содержит только три train-объекта и отсутствует в
validation/test. Поэтому все 2 388 областей объединены в класс `fracture` до начала
обучения.

## Эксперименты

Проверены два профиля:

- **Fast:** YOLO11n, 640 px, 35 эпох, batch 16;
- **Accurate:** YOLO11s, 768 px, 50 эпох, batch 8.

Обе модели инициализированы предобученными весами. Seed равен 42, используется early
stopping. Для X-ray отключены hue/saturation, vertical flip, mosaic и mixup. Остались
небольшие повороты, сдвиги, масштабирование, изменение яркости и horizontal flip.

Гиперпараметры выбирались по validation. Test split использован после фиксации моделей.

| Model | mAP@0.5 | mAP@0.5:0.95 | Precision | Recall | Inference, ms | Size, MB |
|---|---:|---:|---:|---:|---:|---:|
| Fast | 0.1536 | 0.0428 | 0.2985 | 0.1979 | 111.1 | 5.19 |
| Accurate | 0.1487 | 0.0411 | 0.2601 | 0.1868 | 338.7 | 18.27 |

Гипотеза о преимуществе YOLO11s/768 не подтвердилась. Fast немного лучше по всем четырём
quality-метрикам и примерно втрое быстрее. Порог `mAP@0.5 = 0.5` из ТЗ не достигнут.

## Режим повышенной чувствительности

Для Fast добавлен второй inference-проход по двум перекрывающимся полосам. Он запускается,
только если обычный проход не дал результата. Кандидат должен иметь confidence не ниже
`0.28` и пересекаться со слабым сигналом на полном изображении.

Проверка при пользовательском пороге `0.25`:

| Split | Режим | Precision | Recall | F1 |
|---|---|---:|---:|---:|
| Validation | Обычный | 0.434 | 0.162 | 0.236 |
| Validation | Чувствительный | 0.381 | 0.181 | 0.246 |
| Test | Обычный | 0.344 | 0.115 | 0.172 |
| Test | Чувствительный | 0.342 | 0.146 | 0.204 |

На test split режим поднял recall и F1 почти без изменения precision. В интерфейсе он
доступен только для Fast.

## Приложение

```text
Browser → Streamlit → POST /predict → FastAPI → ModelManager → YOLO
                                                ↓
                           detections + annotated PNG
```

Backend проверяет MIME-тип, размер файла и количество пикселей. Модели загружаются при
первом обращении и затем остаются в памяти. Для каждого профиля используется отдельный
inference lock.

Frontend показывает исходный снимок, результат, число областей, latency и таблицу
координат. Ошибки backend и пустые ответы обрабатываются отдельно.

- Frontend: <https://jcvh3mrbarmvekrnoybn6d.streamlit.app/>
- Backend: <https://bone-fracture-detector-xpw8.onrender.com>
- Swagger: <https://bone-fracture-detector-xpw8.onrender.com/docs>

Render работает на бесплатном CPU. В публичных проверках Fast занимал примерно 15–49 с,
Accurate — 31–97 с. Frontend ждёт inference до 300 с, чтобы пережить холодный старт.

## Разбор ошибок

В `reports/figures/` сохранены ground truth и предсказания обеих моделей. Основные случаи:

- небольшие области с низким контрастом;
- пропуски на снимках с нетипичным кадрированием;
- ложные срабатывания рядом с суставами;
- различия между источниками снимков.

Текущего объёма разметки недостаточно для порога из ТЗ. Следующий цикл экспериментов стоит
начать с ревизии bounding boxes и добавления снимков с редкими типами переломов.

## Итог

Backend, frontend, выбор двух моделей, GitHub и публичный деплой собраны в один сервис.
По test-метрикам предпочтителен Fast: он быстрее, меньше и немного точнее. Accurate
сохранён как второй профиль и как результат отдельного эксперимента с размером модели и
разрешением входа.
